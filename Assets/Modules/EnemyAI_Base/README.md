# EnemyAI_Base

## Назначение

`EnemyAI_Base` — reusable-модуль базового поведения врага для прототипов на `NavMeshAgent`.
Он координирует:

- поиск и выбор цели;
- преследование;
- атаку;
- реакцию на получение урона;
- возврат в idle после потери цели.

Модуль не наносит урон напрямую и не зависит от конкретного проекта, но ожидает корректную настройку цели, анимации, движения и атаки.

## Состав модуля

- `EnemyAI_Base` — основной orchestrator поведения.
- `EnemyMovement` — движение к точке / цели.
- `EnemyVision` — проверка видимости.
- `EnemyAnimationController` — мост к `Animator`.
- `EnemyAttack` — нанесение урона.
- `EnemyTargetRegistry` — лёгкий реестр доступных целей.
- `EnemyTargetRegistryMember` — helper-компонент для регистрации целей в реестре.

## Как теперь работает поиск целей

Поиск потенциальных целей идёт в 2 шага:

1. Основной путь: через `EnemyTargetRegistry`.
2. Legacy fallback: через `GameObject.FindGameObjectsWithTag(...)`, если для нужного тега в реестре нет подходящих целей.

Это даёт два преимущества:

- текущий проект не ломается и продолжает работать со старой сценной настройкой;
- новый проект может использовать более чистую и переносимую схему без постоянного полного поиска по сцене.

## Настройка выбора целей

Ключевые поля `EnemyAI_Base`:

- `allowedTargetTags` — основной список допустимых тегов целей.
- `targetPriorityMode` — режим выбора:
  - `TagOrder`
  - `NearestAllowedTarget`
- `lostTargetCooldown` — сколько секунд враг ещё помнит цель после потери видимости.
- `lostTargetCooldown` отсчитывается не от последнего кадра видимости, а от момента, когда враг дошёл до последней известной позиции цели.
- `retargetDistanceAdvantage` — на сколько новая цель должна быть ближе текущей, чтобы враг переключился без дёрганья.
- `playerTag` — legacy fallback, используется только если `allowedTargetTags` пуст.

Правило обратной совместимости:

- если `allowedTargetTags` заполнен, используется он;
- если `allowedTargetTags` пуст, модуль работает по старому пути через `playerTag`.

## Рекомендуемый способ интеграции в новый проект

Для лучшей переносимости:

1. Повесьте `EnemyTargetRegistryMember` на корень игрока или другой targetable-сущности.
2. Убедитесь, что у объекта выставлен нужный Unity `Tag`.
3. Добавьте этот тег в `allowedTargetTags` у enemy prefab.

Пример:

- игрок: тег `Player` + `EnemyTargetRegistryMember`
- мирный житель: тег `RescueTarget` + `EnemyTargetRegistryMember`
- враг: `allowedTargetTags = [Player, RescueTarget]`, `targetPriorityMode = NearestAllowedTarget`

## Совместимость с разными типами целей

Модуль совместим с:

- `Player`
- `RescueTarget`
- любыми другими объектами с подходящим тегом
- объектами, реализующими `ITargetableEntity`

Если на цели есть `ITargetableEntity`, `EnemyAI_Base` дополнительно проверяет `IsValidEnemyTarget`.
Это позволяет автоматически исключать:

- спасённые rescue-target;
- проваленные rescue-target;
- другие цели, которые временно или навсегда стали невалидными.

## Поток поведения

1. Враг периодически ищет допустимую цель.
2. В обычном gameplay-режиме `NearestAllowedTarget` враг выбирает ближайшую валидную цель среди всех разрешённых тегов.
3. Если во время chase рядом появляется другая валидная цель, которая заметно ближе текущей, враг переключается на неё.
4. Если цель видна, враг начинает преследование.
5. Если цель входит в радиус атаки, запускается `EnemyAttack`.
6. Если цель пропадает из видимости, враг продолжает идти к последней известной позиции без немедленного истечения памяти.
7. Когда враг добирается до этой позиции, только тогда запускается таймер `lostTargetCooldown`.
8. Если цель окончательно потеряна, враг пересчитывает доступные цели и только потом возвращается в idle.
9. Если враг получает урон и остаётся жив, после завершения `hit reaction` он может заагриться на источник урона.

## Как работает ближнее обнаружение

`EnemyVision` использует два разных режима обнаружения:

- обычное зрение через `viewDistance`, `viewAngle` и проверку препятствий `Physics.Raycast`;
- ближнее "чувство" через `aggroDistance`.

Важное правило для `aggroDistance`:

- близкая цель больше не считается автоматически обнаруженной только по голой дистанции;
- ближнее обнаружение срабатывает только если до цели есть непрерывный путь по `NavMesh`;
- для этого используется `NavMesh.CalculatePath(...)` и принимается только `PathComplete`.

Это убирает ложный агро-сценарий, когда игрок стоит близко, но отделён стеной, закрытой дверью или несвязанной частью уровня без прохода по `NavMesh`.

## Визуализация в Editor

В редакторе `EnemyVision` всегда рисует gizmo в `Scene View`, пока игра не запущена:

- жёлтая сфера — `aggroDistance`, дистанция ближнего "чувства";
- голубая сфера — дальняя дистанция обычного обзора `viewDistance`.

Во время `Play Mode` gizmo не рисуется, чтобы не засорять игровой вид.

## Рекомендуемый режим для зомби в gameplay

- Для обычного игрового поведения рекомендуется:
  - `allowedTargetTags = [Player, RescueTarget]`
  - `targetPriorityMode = NearestAllowedTarget`
- `TagOrder` стоит оставлять только для специальных сценариев, где нужен жёсткий приоритет цели.

## Что требуется для переноса в другой проект

Минимально:

- `NavMeshAgent` или допустимый fallback movement path;
- `Animator` с совместимыми параметрами и state names;
- хотя бы один валидный target tag;
- `EnemyAttack`, если нужен боевой цикл;
- `EnemyVision`, если нужна реальная проверка видимости.

Желательно:

- использовать `EnemyTargetRegistryMember` на целях;
- не строить новый setup только вокруг `playerTag`;
- явно заполнять `allowedTargetTags`.

## Что остаётся конфигурационной зависимостью

Это нормальные, но реальные зависимости модуля:

- Unity tags целей;
- структура enemy prefab;
- `NavMesh`;
- animator states и trigger-параметры;
- наличие `Health` / `IDamageable` у атакуемых целей.

## Legacy fallback

`playerTag` сохранён для обратной совместимости со старыми prefab и сценами.
Это вторичный путь, а не основной рекомендуемый механизм поиска целей.

Если переносите модуль в новый проект, лучше считать legacy fallback только страховкой, а не основным способом интеграции.
